# nexa 产品定位（正确定义）

> 纠正：nexa **不是**「传统 OA 对接钉钉数据源」。  
> nexa **就是**新一代 **企业钉钉 / 企业协作平台本体**。

## 一句话

**nexa = 可接入的企业钉钉**  
组织 · 通讯录 · 审批 · 待办 · IM · 工作台 · 数据中心 · **企业 Agent（大脑）**  
客户/用户可以 **注册租户、接入使用**，而不是先买一套旧 OA 再去「对接钉钉」。

## 旧方案为什么是垃圾

| 旧路 | 问题 |
|------|------|
| OA 为主，钉钉当数据源同步 | 双系统、主数据打架、体验割裂 |
| 人事/审批在 OA，消息在钉钉 | 人永远在两个 App 里跳 |
| 对接层堆 adapter | 成本高、能力薄、永远跟在钉钉后面 |

## nexa 正确形态

```
用户/企业 ──注册租户──► nexa（本体）
                          │
          ┌───────────────┼────────────────┐
          ▼               ▼                ▼
      组织通讯录        审批/待办         IM/工作台
          │               │                │
          └───────────────┴──── Agent ─────┘
                              │
                    可选：外部业务系统连接器
                    （ordersys / ERP / 财务库 / 任意 HTTP）
                    不是「对接钉钉」，是 nexa 自己接业务
```

### 核心产品面

1. **租户接入（Tenant Onboarding）**  
   企业注册 → 建组织 → 邀成员 → 配权限 → 立即可用  
2. **协作本体**  
   部门/成员、审批流、待办、日历、IM、工作台（像钉钉，不是旁路）  
3. **企业 Agent**  
   对话即入口；技能覆盖 nexa 全部能力 + 外部连接器  
4. **连接器（Connectors）**  
   可选接入外部业务数据（CDC/API），服务分析与导出  
5. **掌上 App**  
   员工日常入口；品牌可白标

### 明确不是

- 给旧 OA 做的「钉钉同步插件」  
- 只能一家公司自用的内部单体  
- 先 Java OA 再套 Agent  

## 与「钉钉」的关系（对外表述）

| 说法 | 对错 |
|------|------|
| nexa 是企业钉钉类产品 | 对 |
| nexa 支持用户/企业接入开通 | 对 |
| nexa 用钉钉 OpenAPI 当唯一主数据源 | 错（可兼容导入，但不是产品定义） |
| 旧 OA 对接钉钉 | 错（废弃思路） |

可选：**兼容模式**——客户若已有钉钉，可导入通讯录；但主路径是 nexa 自有组织与协作。

## 技术栈

- 领域服务 / 网关 / CDC / 数据中心：Go  
- Agent：Node + NeoX  
- 掌上：Flutter  
- 无 Java 运行时  

## 里程碑重心

1. Tenant 接入闭环（注册/邀请/成员）  
2. 组织+审批+待办+IM 作为产品主路径  
3. Agent 默认只认 nexa 技能  
4. 连接器框架（外部业务库/API）  
5. 钉钉 OpenAPI 仅作「导入/兼容」连接器之一  

## 部署形态（省资源）

- **nexa-core** 单端口承载全部业务域 + 网关
- **nexa-iam** 独立认证/租户
- **nexa-agent** 独立对话
